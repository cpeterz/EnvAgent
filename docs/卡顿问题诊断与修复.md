# 运行卡顿问题诊断与修复

## 问题现象

运行环境新闻 Agent 时，流程总是在"摘要循环"阶段卡住，迟迟跑不出结果。日志通常停在如下位置后便不再有新输出：

```
2026-06-14 15:27:06,723 [INFO] [Summarizing] 多地出台支持绿电交易政策-中国循环经济协会
2026-06-14 15:27:07,274 [INFO] [Summarizing] Failed - invalid browsed content
2026-06-14 15:27:07,279 [INFO] [Backup News Rejected] 多地出台支持绿电交易政策-中国循环经济协会
（之后无任何日志，进程挂起）
```

## 根因分析

卡顿发生在每个栏目内部的"逐条新闻摘要"子流程中，定位到三个缺陷：

### 1. `run_with_timeout` 错误使用 `asyncio.shield`（核心 bug）

原实现：

```python
async def run_with_timeout(coro, *, timeout: float, default: T) -> T:
    task = asyncio.ensure_future(coro)
    try:
        return await asyncio.wait_for(asyncio.shield(task), timeout=timeout)
    except (asyncio.TimeoutError, asyncio.CancelledError):
        return default
    except Exception:
        return default
```

- `asyncio.wait_for(asyncio.shield(task), ...)` 超时后，`wait_for` 只取消了外层的 `shield` 包装，**真正的 LLM / 浏览任务并没有被取消**，会变成永远运行的"孤儿任务"。
- 孤儿任务持续占用 agently 的网络连接 / 会话不释放。一旦 LLM 端点偶尔变慢，孤儿任务越积越多，后续调用排队等待连接，整体表现为"越跑越卡"。

### 2. 吞掉 `CancelledError`

同一函数把 `asyncio.CancelledError` 当作普通超时返回 `default`。当上层 `for_each` 想取消某个 worker 时，任务"拒绝被取消"，导致 `gather` 永远等不到它结束，整条子流程挂死，且不打印任何日志——正好对应日志在 merge 之后突然静止的现象。

### 3. 浏览（browse）的 httpx 未设置超时

`tools/browse.py` 创建 agently `Browse` 工具时没有传 `timeout`，agently 内部 `_bs4_browse` 使用 `timeout=None`，即 httpx **无超时**。虽然外层有 20 秒 `wait_for` 兜底，但其依赖任务取消机制，与上面的 bug 叠加后并不可靠。

## 修复方案

### 修复 1+2：重写 `run_with_timeout`（`workflow/common.py`）

```python
async def run_with_timeout(coro, *, timeout: float, default: T, label: str = "") -> T:
    # 不再用 asyncio.shield 包裹 coro：wait_for 必须能在超时时取消底层协程，
    # 否则 LLM / browse 任务会变成孤儿持续占用网络连接，逐步拖垮整个流程。
    start = time.monotonic()
    try:
        return await asyncio.wait_for(coro, timeout=timeout)
    except asyncio.TimeoutError:
        _diag_logger.warning(
            "[Timeout] %s exceeded %.0fs (waited %.1fs); task cancelled.",
            label or "async-call", timeout, time.monotonic() - start,
        )
        return default
    except asyncio.CancelledError:
        # 绝不吞掉取消信号：重新抛出，让外层 for_each / gather 能真正回收任务。
        raise
    except Exception as exc:
        _diag_logger.warning("[AsyncError] %s: %s", label or "async-call", exc)
        return default
```

- 去掉 `asyncio.shield`，超时会真正取消底层任务，不再产生孤儿。
- 重新抛出 `CancelledError`，让上层正常回收 worker，避免子流程挂死。
- 增加 `label` 参数，超时 / 异常时打印可定位的日志。

### 修复 3：为 browse 设置硬超时

- `news_collector/config.py` 的 `BrowseConfig` 新增 `timeout` 字段（默认 15 秒，可在 `SETTINGS.yaml` 的 `BROWSE.timeout` 配置）。
- `tools/browse.py` 在创建 `Browse` 时传入 `timeout=settings.browse.timeout`，让 httpx 层也有硬超时，不再依赖任务取消。

### 配套：增加诊断日志（`workflow/summary_chunks.py`）

便于再次运行时精确定位瓶颈所在：

| 日志标记 | 含义 |
| --- | --- |
| `[Dispatch] cursor=.. batch=.. target=.. results=.. candidates=..` | 每个摘要批次的派发状态 |
| `[Summary Loop Continue]` / `[Summary Loop Done]` | 摘要循环的每次决策（覆盖原先静止的位置） |
| `[Browsed] x.xs len=N` | 每次网页抓取的耗时与内容长度 |
| `[Summary LLM] x.xs` | 每次摘要 LLM 调用耗时 |
| `[Timeout] <label>` / `[AsyncError] <label>` | LLM 调用超时 / 异常，带调用标签 |
| `browse TIMEOUT` / `browse error` | 抓取超时（WARNING）或异常，区分两种情况 |

其余 LLM 调用（`pick_news`、`write_column`、`generate_outline`）也补充了 `label`，超时日志可直接识别。

## 涉及的文件

- `workflow/common.py`：重写 `run_with_timeout`，新增诊断 logger。
- `workflow/summary_chunks.py`：增加 browse / LLM 耗时与摘要循环诊断日志。
- `workflow/column_chunks.py`：`write_column` 调用补充 `label`。
- `workflow/report_chunks.py`：`generate_outline` 调用补充 `label`。
- `news_collector/config.py`：`BrowseConfig` 新增 `timeout` 字段。
- `tools/browse.py`：向 agently `Browse` 传入 `timeout`。

## 再次运行时的排查指引

根据日志最后停留的标记定位剩余瓶颈：

- 停在某条 `[Browsed]` 之后没有 `[Summary LLM]`，并出现 `[Timeout] summarize:...` → LLM 端点慢 / 卡，属于模型服务问题（也解释了"别人三分钟"——他们的端点更快）。
- 停在 `[Summary Loop Continue]` 之后没有 `[Dispatch]` → agently 子流程事件派发层面的问题。
- 大量 `[Browsed] 20.0s` 或 `browse TIMEOUT` → 网络 / 代理抓取慢，需调小 `BROWSE.timeout` 或配置代理。

# 第二轮：摘要子流程事件循环死锁（根因）

## 问题现象

在完成第一轮修复后，再次运行仍会卡住，但卡点不同。日志精确停在某次 `[Dispatch]` 之后、且**没有任何 `[Summarizing]` 行**：

```
2026-06-15 14:50:33,376 [INFO] [Summary Loop Continue] results=3 target=4 cursor=7 candidates=8
2026-06-15 14:50:33,378 [INFO] [Dispatch] cursor=7 batch=1 target=4 results=3 candidates=8
（之后无任何日志，进程挂起；本次运行未产出报告文件）
```

同时日志中夹杂大量模型重试警告：

```
WARNING:  [ModelResponseResult] [model.retrying] No target data in response. Preparing retry.
WARNING:  [AsyncError] pick_news: Can not satisfy ensure keys [...] within 3 retries.
```

## 根因分析

### 1. 事件驱动的摘要循环在多次迭代后死锁（核心 bug）

原摘要子流程是一个"事件循环状态机"：

```
prepare → emit Summary.Dispatch
when("Summary.Dispatch") → dispatch_batch → for_each(concurrency) → summarize → merge
merge → 再 emit Summary.Dispatch（或 Summary.Done）
```

且该子流程是**嵌套在栏目子流程里的二级 `to_sub_flow`**，内部又反复 `emit` 同一个 `Summary.Dispatch` 事件、反复触发同一个 `for_each`。

- 日志证据：第 1/2/3 次 `Dispatch` 都正常 fan-out 并产出 `[Summarizing]`，但第 4 次 `Dispatch cursor=7` 之后 `for_each` **再也没有派发** worker，`summarize_candidate` 根本没被进入（连第一行 `[Summarizing]` 都没有）。
- 说明卡点不在我们的业务代码，而在 agently 的「嵌套子流程 + 反复重新 emit 事件 + 反复 `for_each`」组合上——多次迭代后事件派发死锁。
- 模型频繁返回 `No target data in response` 触发重试只是放大了时序，让这个死锁更容易被触发，并非根因。

### 2. 模型端点退化（次要诱因，非代码问题）

`No target data in response. Preparing retry` 表示 LLM 端点返回了空 / 不合规的结构化输出，agently 默认重试 3 次、每次约 5 秒。端点状态好时每次调用 2-3 秒完成；退化时每次结构化调用要耗 ~20 秒在重试上，叠加多栏目多新闻后整体变得极慢。这部分属于模型服务侧问题，可通过降低并发、检查端点限流缓解。

## 修复方案

**用一个自包含的有界并发异步段，替换掉整套事件循环状态机。**

删除以下事件驱动结构：

- `build_summary_sub_flow`（含 `Summary.Dispatch` / `Summary.Done` 事件与嵌套 `for_each`）
- `prepare_summary_candidates` / `dispatch_summary_batch` / `summarize_candidate` / `merge_summary_batch` / `finalize_summary` 五个 chunk

替换为单个 `create_summarize_column_chunk`（`workflow/summary_chunks.py`），在一个普通 `while` 循环里用 `asyncio.gather` 做有界并发摘要，达到目标条数即停止：

```python
while len(summarized_news) < target_count and cursor < len(candidates):
    batch_size = min(concurrency, target_count - len(summarized_news), len(candidates) - cursor)
    batch = candidates[cursor:cursor + batch_size]
    cursor += batch_size
    summaries = await asyncio.gather(
        *(summarize_single_news(config, logger, browse_tool, column_outline, c["news"]) for c in batch),
        return_exceptions=True,
    )
    for candidate, summary in zip(batch, summaries):
        if isinstance(summary, dict):
            summarized_news.append(summary)
        elif isinstance(summary, BaseException):
            logger.warning("[Summary Worker Error] %s: %s", news_title, summary)
        elif candidate.get("is_backup"):
            logger.info("[Backup News Rejected] %s", news_title)
        else:
            logger.info("[Backup News Activated] %s", news_title)
```

栏目子流程相应简化为线性串联（`workflow/env_news.py`）：

```python
flow.to(search_column_news).to(pick_column_news).to(summarize_column_news).to(write_column).end()
```

该方案：

- 彻底移除"反复 emit 事件 + 嵌套 for_each"的死锁来源，逻辑也比原 5-chunk 状态机更简单。
- 完整保留原有语义：候选优先级（picked 在前、backup 在后）、目标条数提前停止、并发度、备用激活 / 拒绝日志。
- 备用兜底（`[Backup News Activated]` / `[Backup News Rejected]`）与并发度（`WORKFLOW.summary_concurrency`）行为不变。

## 涉及的文件（第二轮）

- `workflow/summary_chunks.py`：删除 5 个事件驱动 chunk 工厂与未用辅助函数，新增 `create_summarize_column_chunk`，更新 `__all__`。
- `workflow/env_news.py`：删除 `build_summary_sub_flow`，栏目子流程改为线性串联到 `summarize_column_news`。

## 缓解模型重试风暴的建议

- 端点出现大量 `No target data in response` 时，适当调小 `WORKFLOW.summary_concurrency`（如 3 → 1）以减轻端点限流压力。
- 确认 `LLM_BASE_URL` 指向的服务未被限流 / 未过载；必要时更换更稳定的模型端点。
