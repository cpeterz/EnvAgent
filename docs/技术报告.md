# 基于多Agent协作的环境新闻智能采集与报告生成系统

## 摘要

本文提出了一种面向环境领域的新闻智能采集与报告自动生成系统（Env News Agent）。系统针对中国大陆网络环境下主流搜索引擎不可用、环境新闻源分散、人工筛选效率低等痛点，设计了多引擎自适应搜索轮换、三阶段渐进式内容过滤、双层编辑Agent协作、事件驱动可恢复工作流以及领域专用提示词工程等五项核心创新。系统基于Agently框架构建，采用TriggerFlow事件驱动架构实现三级嵌套工作流编排，通过Protocol-based工具抽象层整合搜索引擎、网页浏览器和RSS源三类数据采集通道，并以FastAPI + SSE提供实时交互式Web界面。实验表明，系统在"碳排放政策""大气污染防治""环保督察"三个环境主题上均能自动完成从信息采集到专业报告生成的全流程，输出的新闻摘要专业准确、推荐理由具有领域针对性，且在搜索引擎限流条件下仍能通过降级策略保持有效输出。

**关键词**：环境新闻采集；多Agent协作；大语言模型；事件驱动工作流；搜索引擎轮换

## 1 引言与背景

### 1.1 研究动机

环境新闻具有信息源高度分散、专业性要求高、时效性敏感的特点。环境从业者、研究人员和政策制定者需要持续跟踪来自政府公报、行业媒体、国际组织和地方新闻等多渠道的环境动态，涵盖碳排放政策、大气污染防治、生态保护、环保督察等多个子领域。传统的人工新闻筛选方式面临三大挑战：（1）信息碎片化——环境新闻散布于数十个官方网站、行业门户和国际机构主页，人工逐一访问效率极低；（2）专业门槛高——判断一则新闻的环境政策意义、生态影响和行业关联需要领域知识，通用新闻聚合工具难以胜任；（3）时效性压力——环境政策和监管动态更新频繁，每日简报的生成窗口仅有数小时。

### 1.2 现有方案的局限

当前的新闻聚合方案主要存在以下不足：

- **通用新闻聚合器**（如Google News、今日头条等）面向大众消费场景，缺乏环境领域的专业筛选能力，无法区分政策法规文件与一般性报道的重要性差异。
- **搜索引擎API方案**（如DuckDuckGo、Bing等）在中国大陆网络环境下可用性受限——DuckDuckGo无法直接访问，Bing中国版频繁重定向至首页，百度对服务器端请求实施严格的验证码拦截。
- **RSS订阅方案**虽然稳定，但环境领域的高质量RSS源数量有限，且无法覆盖突发性环境事件和地方新闻。

### 1.3 本文贡献

针对上述问题，本文提出Env News Agent系统，主要贡献包括：

1. 设计了面向中国大陆网络环境的多引擎自适应搜索轮换机制，在搜狗和360搜索之间实现有状态的负载均衡与阻断恢复。
2. 提出了"搜索—筛选—摘要"三阶段渐进式内容过滤流水线，在召回率和精确度之间实现梯度平衡。
3. 构建了模拟新闻编辑室分工的双层Agent协作架构，通过总编Agent和栏目编辑Agent的角色分离提升报告的结构性和专业性。
4. 基于Agently TriggerFlow实现了三级嵌套的事件驱动工作流，支持并发控制、超时保护和失败降级。
5. 设计了环境领域专用的YAML提示词模板体系，将领域知识系统性地注入Agent决策过程。

## 2 系统架构与核心方法

### 2.1 整体架构

系统采用四层架构设计：

```
┌─────────────────────────────────────────────────────────┐
│                    Web界面层 (FastAPI + SSE)              │
│    主题输入 → 实时进度展示 → 报告渲染 → 历史管理          │
├─────────────────────────────────────────────────────────┤
│                  业务逻辑层 (EnvNewsCollector)             │
│    配置管理 / 日志系统 / Agently Agent创建与调度           │
├─────────────────────────────────────────────────────────┤
│              工作流编排层 (TriggerFlow)                    │
│    主流程 → 栏目子流程(×N) → 摘要子流程(×M)               │
│    事件驱动 / 并发控制 / 超时保护 / 降级Fallback          │
├─────────────────────────────────────────────────────────┤
│                    工具抽象层 (Protocol)                   │
│  SearchTool (Sogou+360)  │  BrowseTool (BS4)  │  RSSFeed │
└─────────────────────────────────────────────────────────┘
```

- **工具抽象层**定义了`SearchToolProtocol`、`BrowseToolProtocol`和`RSSFeedToolProtocol`三个协议接口，使上层业务逻辑与具体数据源实现解耦。
- **工作流编排层**通过Agently TriggerFlow构建三级嵌套流程图，以事件（`Summary.Dispatch`、`Summary.Done`等）驱动流程推进。
- **业务逻辑层**负责配置加载、Agent实例创建、工具注入和流程启动。
- **Web界面层**提供FastAPI HTTP服务和SSE实时进度推送，支持浏览器端交互操作。

### 2.2 核心创新一：多引擎自适应搜索轮换

在中国大陆网络环境下，DuckDuckGo、Bing和百度均无法稳定用于服务器端新闻搜索。系统选择搜狗和360搜索作为双引擎，并设计了以下自适应轮换机制：

**有状态引擎管理**：每个搜索引擎封装为独立的`_EngineSession`对象，维护各自的HTTP客户端、阻断状态和阻断时间戳。当引擎返回包含`antispider`或`qcaptcha`标记的响应时，系统将该引擎标记为阻断状态并关闭其连接，300秒冷却期后自动恢复可用。

**请求节流与轮换**：系统维护全局最小请求间隔（4秒），每次搜索前检查距上次请求的时间差，必要时主动休眠。引擎按轮询顺序选择，跳过处于阻断状态的引擎。

**真实URL提取**：搜狗搜索结果中的链接通常为`/link?url=`格式的重定向地址，需要额外的HTTP请求才能获取真实URL。系统直接从HTML的`data-url`数据属性中提取目标网址，避免了重定向请求带来的延迟和失败风险。360搜索同理，通过`data-mdurl`属性提取，并过滤掉`https://www.so.com/link`格式的中转链接。

**线程安全**：搜索操作通过`threading.Lock`保护，确保多栏目并行搜索时引擎状态的一致性。异步接口通过`asyncio.to_thread`桥接同步搜索逻辑。

### 2.3 核心创新二：三阶段渐进式内容过滤

系统将新闻从原始搜索结果到最终报告的筛选过程分为三个精度递增的阶段：

**第一阶段——搜索采集（高召回）**：通过搜索引擎和RSS源并行采集候选新闻，以标题和摘要片段为基本单元。每个栏目生成至多2组搜索查询（关键词组合 + 主题词提取），每组返回至多10条结果。RSS源通过关键词匹配进行初步过滤。所有结果经URL去重后合并为统一格式（`{id, title, brief, url, source, date}`）。

**第二阶段——LLM智能筛选（平衡精度与召回）**：栏目编辑Agent对候选列表执行结构化评估，为每条新闻输出`can_use`（布尔值）、`relevance_score`（0-100整数）和`recommend_comment`（推荐理由文本）。系统按相关性得分降序排列，选取得分最高的若干条（上限由`max_news_per_column`配置控制）进入下一阶段。未被选中但具有一定相关性的新闻作为备用候选保留。

**第三阶段——全文浏览与摘要生成（高精度）**：对入选新闻逐条执行网页全文浏览（通过BeautifulSoup或Playwright提取正文内容），随后由栏目编辑Agent基于全文生成结构化摘要。系统对浏览结果实施严格质量检测——内容长度低于80字符、包含验证码标记（captcha、access denied等8类无效标记）的页面将被拒绝。当主选新闻的全文浏览或摘要失败时，系统自动从备用候选队列中调度替补，直至达到目标数量或候选耗尽。

### 2.4 核心创新三：双层编辑Agent协作

系统模拟真实新闻编辑室的分工模式，设计了两类Agent角色：

**总编Agent（Chief Editor）**：负责报告整体架构设计。接收用户输入的新闻主题后，生成包含报告标题和栏目列表的结构化大纲，每个栏目包含标题（`column_title`）、内容要求（`column_requirement`）和搜索关键词（`search_keywords`）。总编Agent的系统提示词设定其为"资深环境新闻总编辑"，具备环境政策、生态保护、气候变化、污染治理和碳排放交易等领域的深度知识。

**栏目编辑Agent（Column Editor）**：负责单个栏目内的新闻筛选、摘要生成和栏目撰写三项任务。其系统提示词设定其为"细致的环境新闻编辑"，能够识别新闻事件的环境科学意义和政策框架关联。栏目编辑Agent在三个环节分别使用不同的YAML提示词模板（`pick_news.yaml`、`summarize_news.yaml`、`write_column.yaml`），确保每个环节的指令精确匹配其任务目标。

这种角色分离使得总编关注宏观结构的合理性，栏目编辑专注微观内容的准确性，避免了单一Agent在复杂任务中的注意力稀释。同时，支持系统通过`SETTINGS.yaml`中的`use_customized`选项绕过大纲生成、直接使用预设栏目，为特定应用场景提供确定性输出。

### 2.5 核心创新四：事件驱动可恢复工作流

系统基于Agently TriggerFlow框架构建了三级嵌套的事件驱动工作流：

```
主流程：prepare_request → generate_outline → for_each(column) → render_report → end
                                                    │
                                             栏目子流程
                                                    │
            search_news → pick_news → [摘要子流程] → write_column → end
                                           │
                                    摘要子流程（事件循环）
                                           │
            prepare → dispatch → for_each(summarize) → merge
                        ↑                                │
                        └── Summary.Dispatch ←───────────┘
                                                         │
                                              Summary.Done → finalize → end
```

**事件驱动调度**：摘要子流程通过`Summary.Dispatch`和`Summary.Done`两个事件实现批处理循环。每一轮调度一批候选新闻进行并行摘要（并发度由`summary_concurrency`控制），合并结果后根据已完成数量决定继续调度还是终结。

**并发控制**：栏目子流程通过`for_each(concurrency=N)`控制并行处理的栏目数量。在当前配置中，栏目处理设置为串行（`column_concurrency=1`）以降低搜索引擎压力，摘要处理设置为3路并发（`summary_concurrency=3`）以提高浏览效率。

**超时保护**：所有LLM调用均封装在`asyncio.wait_for`中，大纲生成和栏目撰写设置60秒超时，新闻摘要设置45秒超时，筛选评分设置60秒超时。超时后触发降级逻辑而非直接失败。

**容错级联**：当LLM调用超时或返回无效结果时，系统执行多级降级——栏目撰写失败时使用模板化导语和原始摘要组装栏目内容（`_build_fallback_column`）；新闻摘要失败时从备用候选队列中调度替补；大纲生成支持切换到预设自定义大纲。

### 2.6 核心创新五：领域专用提示词工程

系统设计了四套YAML格式的提示词模板，将环境领域知识系统性地注入Agent决策：

| 模板文件 | 功能 | 领域定制要点 |
|---------|------|------------|
| `create_outline.yaml` | 大纲生成 | 预置环境领域栏目方向（政策法规/污染治理/气候变化/生态保护），限制最大栏目数量 |
| `pick_news.yaml` | 新闻筛选 | 要求评估环境相关性得分，输出结构化的`can_use`+`relevance_score`+`recommend_comment` |
| `summarize_news.yaml` | 新闻摘要 | 要求提取环境影响、政策意义等关键信息，输出`can_summarize`判断和结构化摘要 |
| `write_column.yaml` | 栏目撰写 | 生成专业导语，为每条新闻精炼推荐理由，使用新闻编辑的专业语言 |

所有模板通过`mappings`字典注入运行时变量（主题、日期、语言、栏目要求等），Agent通过`load_yaml_prompt`方法加载模板并执行。输出结果通过`ensure_keys`机制强制校验结构完整性（如`column_list[*].column_title`确保每个栏目都有标题），避免LLM输出格式不一致导致的下游解析错误。

## 3 关键技术实现

### 3.1 搜索引擎HTML解析

系统使用BeautifulSoup对搜狗和360搜索的HTML响应进行结构化解析：

- **搜狗解析策略**：以`div.vrwrap`为搜索结果容器，从`h3 a`提取标题，通过`[data-url]`选择器获取真实目标URL（优先级高于`href`属性中的重定向链接），从`div.text-layout p`、`p.star-wiki`等多个候选选择器中提取摘要片段（要求长度大于15字符），从`cite`元素提取来源信息。
- **360解析策略**：以`li.res-list`为结果容器，从`h3 a`的`data-mdurl`属性或`[data-mdurl]`子元素提取真实URL，过滤以`https://www.so.com/link`开头的中转链接，从`p.res-desc`提取摘要。

### 3.2 Protocol-based工具抽象

系统定义了三个`typing.Protocol`接口，实现工具层的开放式扩展：

- `SearchToolProtocol`：定义`search_news(query, timelimit, max_results)`异步接口
- `BrowseToolProtocol`：定义`browse(url)`异步接口
- `RSSFeedToolProtocol`：定义`fetch_feeds(category)`异步接口

当前实现分别为`MultiEngineSearchTool`（基于httpx + BeautifulSoup）、`AgentlyBrowseTool`（基于Agently内置Browse工具）和`RSSFeedTool`（基于feedparser）。上层工作流通过`data.require_resource("search_tool")`获取工具实例，完全不依赖具体实现类。

### 3.3 超时保护与容错级联

系统在所有外部调用点（LLM、网页浏览）实施超时保护，形成三层容错体系：

1. **LLM调用超时**：`asyncio.wait_for`包装所有`async_start()`调用，超时值45-60秒。超时后不抛出异常，而是返回空结果或触发降级逻辑。
2. **内容质量门控**：浏览器返回的网页内容经过8类无效标记检测（验证码页面、访问拒绝、内容过短、反爬提示等），不合格内容立即丢弃。
3. **模板降级**：当LLM无法生成栏目导语时，系统使用`_build_fallback_prologue`函数基于栏目标题和新闻标题列表组装模板化导语，确保输出不为空。

### 3.4 备用候选新闻调度

摘要阶段采用主选-备用双队列机制。`build_summary_candidates`函数首先将LLM筛选通过的新闻（`picked_news`）排入主队列（`is_backup=False`），然后将搜索结果中未被选中但URL不重复的新闻排入备用队列（`is_backup=True`）。摘要批处理循环中，当主选新闻浏览失败时，系统自动向后移动游标调度备用新闻，直至摘要完成数量达到目标值（`target_count`）或所有候选耗尽。

### 3.5 SSE实时进度推送

Web端采用Server-Sent Events（SSE）实现采集进度的实时推送：

1. 采集任务启动时，系统动态创建一个`logging.Handler`实例并附加到全局日志器。
2. 该Handler将每条日志记录（时间戳、级别、消息）追加到任务的`progress`列表中。
3. SSE端点以500ms轮询间隔检查`progress`列表的增量，将新增条目作为`progress`事件推送至客户端。
4. 任务完成或失败时，推送`done`事件并携带最终结果或错误信息。

这种设计将工作流内部的日志输出透明地映射为前端可视化进度，无需修改工作流代码即可实现进度展示。

## 4 实验与结果

### 4.1 实验设置

- **语言模型**：gpt-5.4-mini（通过OpenAI Compatible接口调用），temperature=0.2
- **搜索引擎**：搜狗 + 360搜索双引擎轮换
- **并发配置**：column_concurrency=1（串行处理栏目），summary_concurrency=3（3路并行摘要）
- **参数设置**：max_column_num=3，max_news_per_column=4，搜索每组最多10条结果，浏览最小内容长度80字符
- **RSS源**：生态环境部、中国环境报、UN Environment（3个源）

### 4.2 测试结果

系统在三个不同环境主题上进行了端到端测试，结果如下：

| 测试主题 | 生成栏目数 | 有效栏目数 | 新闻条数 | 报告大小 | 报告标题 |
|---------|----------|----------|---------|---------|---------|
| 碳排放政策 | 1 | 1 | 3 | 5,540B | 碳排放政策日报：全国碳市场扩围、地方配套细则与企业减排压力同步升温 |
| 大气污染防治 | 4 | 2 | 4 | 7,617B | 大气污染防治日报：政策执法、重点区域治理与减排进展 |
| 环保督察 | 1 | 1 | 1 | 2,161B | 环保督察日报：政策整改、地方落实与典型问题追踪 |

### 4.3 内容质量分析

**摘要专业性**：以"碳排放政策"主题为例，系统生成的新闻摘要能够准确提取政策文件的核心指标（如"单位地区生产总值能耗下降14.5%""碳债券融资规模力争达到100亿元""核证总规模近700万吨"），并将其置于碳达峰、碳市场和碳金融的政策语境中进行解读，展现出环境领域专业知识的有效注入。

**推荐理由针对性**：每条新闻的推荐理由（`recommend_comment`）不是泛泛的新闻评述，而是具有环境政策分析视角的专业点评。例如，对河北降碳产品项目的推荐理由中指出"地方正在把'可核证、可交易、可激励'机制嵌入双碳治理"，准确捕捉了该政策的制度创新意义。

**导语概括能力**：栏目导语能够从多条新闻中提炼出统一的政策趋势。如"地方碳排放政策落地"栏目的导语概括了"碳排放双控、重点行业减排、碳市场配套和监管执法"四个维度，与该栏目下三条新闻的内容高度吻合。

### 4.4 系统韧性

在"大气污染防治"测试中，系统规划了4个栏目但仅有2个栏目成功获取新闻内容，另外2个栏目因搜索引擎限流导致搜索结果为空。系统通过`render_markdown`中的空栏目过滤机制（跳过标题或新闻列表为空的栏目），仍然输出了内容完整的2个有效栏目，总报告大小达到7,617字节，为三次测试中最大。这表明系统的降级策略能够有效应对部分搜索失败的场景，以"尽力输出"的原则保障报告的可用性。

## 5 讨论

### 5.1 当前局限

**搜索引擎限流**是系统面临的主要瓶颈。搜狗和360搜索在连续请求8-10次后会触发验证码拦截，尽管系统通过4秒最小间隔、引擎轮换和串行栏目处理等策略缓解了这一问题，但在栏目数较多时仍可能出现部分栏目搜索失败。当前的300秒冷却期在单次采集任务中通常无法恢复，需要依赖跨任务的状态持久化。

**RSS源覆盖有限**：系统内置了3个环境领域RSS源（生态环境部、中国环境报、UN Environment），但部分源站的RSS服务不够稳定，且RSS内容通常滞后于搜索引擎索引。RSS作为搜索引擎的补充数据源，其价值在长期稳定运行中更为显著。

### 5.2 改进方向

1. **扩展搜索引擎池**：引入更多中国大陆可用的搜索引擎（如神马搜索、头条搜索），增加轮换池深度。
2. **搜索结果缓存**：对同一查询词在短时间内的重复搜索实施本地缓存，减少引擎请求次数。
3. **自定义栏目模板**：允许用户通过Web界面自定义栏目结构和搜索关键词，绕过大纲生成环节以获得确定性输出。
4. **多模型适配**：当前系统通过OpenAI Compatible接口支持多种模型（GPT、DeepSeek、Claude等），后续可针对不同模型的输出特点调整提示词和解析策略。
5. **定时采集**：增加定时任务功能，实现环境新闻日报的自动化定期生成与推送。

## 6 结论与展望

本文提出的Env News Agent系统成功实现了环境新闻的自动化端到端采集与专业报告生成。通过多引擎自适应搜索轮换解决了中国大陆网络环境下搜索引擎可用性问题，三阶段渐进式内容过滤在信息召回与精确度之间取得了有效平衡，双层编辑Agent协作架构赋予了系统新闻编辑室级别的内容组织能力，事件驱动的可恢复工作流确保了长链路任务的稳定执行，领域专用提示词工程使得输出内容具备环境领域的专业深度。

实验验证表明，系统在碳排放政策、大气污染防治、环保督察三个典型环境主题上均能生成结构完整、内容专业的新闻简报，且在面临搜索引擎限流等外部不确定性时展现出良好的韧性。

未来工作将围绕搜索引擎池扩展、定时采集自动化、多语言输出支持和用户自定义栏目等方向持续优化，进一步提升系统在环境新闻领域的实用价值。
