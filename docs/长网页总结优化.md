# 长网页总结优化：加速 + 降低失败丢条

## 1. 背景与现象

EnvAgent 在"浏览 + 摘要"阶段逐条抓取新闻网页正文，再交给 LLM 生成摘要。
实际运行中发现：**网页内容很长时，总结很慢且经常失败，导致整条新闻被丢弃。**

相关代码：`workflow/summary_chunks.py` 的 `summarize_single_news()`。

## 2. 根因分析

1. **整页内容一次性送入 LLM**
   抓取阶段最多保留 `max_content_length = 12000` 字符（`SETTINGS.yaml` 的 `BROWSE` 段），
   这些字符会原封不动塞进 `summarize_news.yaml` 提示词。输入越长，LLM 处理越慢。
2. **长网页噪声占比高**
   政府门户、新闻站点的长页面里，导航、菜单、页脚、版权、"相关推荐""分享到"等噪声占比大，
   真正正文占比低。LLM 需在大量噪声中定位正文，进一步拖慢速度、增加出错概率。
3. **超时即整条丢弃**
   摘要 LLM 调用有 `timeout=45`，超时后 `run_with_timeout` 返回 `default=None`，
   原代码遇到 `None` 直接 `return None`，该新闻被丢弃。长页面经常逼近甚至超过该超时。
4. **截断方式粗糙**
   抓取阶段的 `max_content_length` 是简单尾部截断，不保证保留正文核心，
   可能把正文截掉、只留页头噪声。

## 3. 解决方案

核心思路：**缩短送入 LLM 的输入 + 智能截断 + 失败兜底**，不增加 LLM 调用次数。

### 3.1 正文提取与智能截断（纯 Python，无额外 LLM 调用）

在 `summarize_single_news()` 调用 LLM **之前**，新增 `_extract_main_content()` 预处理抓取内容：

- **逐行去噪** `_clean_lines()`：去除重复行、丢弃命中噪声关键词（版权 / 备案 / 登录 / 首页 /
  相关推荐 / 分享到 等）的短行、合并连续空行。
- **按边界截断** `_truncate_on_boundary()`：裁剪到上限时优先在段落空行处截断，
  其次在句末标点（。！？.）处截断，避免硬切句子。
- **安全回退**：若去噪后内容低于 `min_content_length`，回退到对原始内容的边界截断，
  避免误删导致内容过短被丢弃。

### 3.2 独立的摘要输入长度上限

新增配置项 `summary_max_content_length`（默认 **6000**），**独立于**抓取阶段的
`max_content_length`（12000）：

- 抓取阶段仍可保留较完整内容；
- 但送入 LLM 的长度被单独压缩到 6000，加快总结、降低超时率。

可在 `SETTINGS.yaml` 的 `BROWSE` 段覆盖；未设置时使用默认 6000，并有 `max(..., 1000)` 下限保护。

### 3.3 失败兜底：搜索 brief 降级（叠加在现有"调备用候选"之上）

原逻辑：摘要失败 → `return None` → 由现有备用候选队列补一条。备用耗尽时该栏目新闻数会减少。

改动：在**超时 / 出错 / 空摘要**分支改为调用 `_build_brief_fallback()`：

- 若新闻自带搜索摘要片段 `brief`，经轻量清洗后（截断到 300 字符内）生成降级摘要返回，
  避免整条丢失；
- 若无 `brief`，仍 `return None`，维持现有调备用逻辑。

> **质量优先用备用候选，数量兜底才用 brief。** 模型主动判定"内容不相关"
> （`can_summarize=False`）时仍 `return None`，不做 brief 兜底，避免把不相关内容塞进报告。

## 4. 相比初版设计修正的缺点

本次迁移在 `old-EnvAgent-long-summary` 初版基础上修正了以下问题：

| # | 初版问题 | 本次修正 |
| --- | --- | --- |
| 1 | **中文标题"标题相关行保护"失效**：`title.split()` 对无空格中文标题只得到整条标题，`term in line` 几乎不命中。 | 新增 `_title_terms()`：中文标题用 2–4 字滑窗切词，英文按空白/标点分词，保护真正生效。 |
| 2 | **去重与标题保护冲突**：初版"标题相关行豁免去重"会让重复标题行被保留多次。 | 去重**始终生效**；标题相关仅豁免"噪声关键词短行"过滤，不豁免精确去重。 |
| 3 | **brief 兜底未清洗**：直接把 `brief` 当摘要返回，可能带噪声。 | `_build_brief_fallback()` 先经 `_clean_lines()` 清洗再返回。 |
| 4 | **删除了诊断日志**：初版丢掉了主目录的耗时/长度日志。 | 保留主目录 `[Browsed] %.1fs len=`、`[Summary LLM] %.1fs`、`label=` 等可观测性日志。 |
| 5 | **异常捕获冗余**：`except (asyncio.TimeoutError, Exception)`。 | 沿用主目录写法，分别处理 `TimeoutError` / `Exception` 并带耗时日志。 |

## 6. 验证方法

1. 启动 Web 服务，触发一次"今日环境新闻"采集。
2. 观察日志 `[Summarizing]` / `[Browsed]` / `[Summary LLM]`：
   - 长网页应不再频繁出现 `Failed - timeout or error`；
   - 出现 `Fallback - using search brief` 表示 brief 兜底生效；
   - `[Summary LLM] %.1fs` 显示单条 LLM 耗时应明显下降。
3. 相同主题下，改动后成功摘要的条数应增加、整体报告生成时间下降。
4. 边界用例：超长页面应在超时内完成或正确降级；抓取失败页面仍走 `None` → 调备用，无报错。
5. `SETTINGS.yaml` 未设 `summary_max_content_length` 时使用默认 6000，设置后生效。

纯函数已做隔离单测验证：中文/英文分词、去噪去重、句末边界裁剪、长内容裁剪到上限、
短内容透传、brief 兜底（有/无 brief、超长截断、噪声清洗）均符合预期。

## 7. 取舍说明

考虑过的替代方案 **分块总结（Map-Reduce）**：将长内容切块分别总结再合并。
未采用，因为它会**显著增加 LLM 调用次数和总耗时**，与本次"加速"目标冲突；
而绝大多数新闻页面的有效正文在去噪 + 6000 字符上限内已足够生成高质量摘要。

> 上线后建议关注日志中 `Fallback - using search brief` 的占比：若偏高，
> 可进一步下调 `summary_max_content_length`（如 4000）配合 brief 兜底。
